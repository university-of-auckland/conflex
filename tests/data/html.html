Overview

Shibboleth is a standards based, open source software package for web single sign-on across or within organisational boundaries. It allows sites to make informed decisions for individual access of protected online resources in a privacy-preserving manner.

The Shibboleth software implements widely used federated identity standards, principally the OASIS Security Assertion Markup Language (SAML), to provide a federated single sign-on and attribute exchange framework. A user authenticates with his or her organizational credentials, and the organisation (or identity provider) passes the minimal identity information necessary to the service provider to enable an authorisation [is the user authorised to access the site] decision. Shibboleth also provides extended privacy functionality allowing a user and their home site to control the attributes released to each application.

Shibboleth IDP3 has been implemented as the identity provider (IdP) for all web-sso protected applications within the university.

The University of Auckland's web-sso provides seamless access to shared computing resources and collaboration environments provided through the Tuakiri access federation

Access and Security

1. Application Security Overview
Shibboleth provides authentication for all Web SSO enabled applications.  It does not provide for group-based authorisations. It is solely an authentication solution.

2. Who are the users?

End users – anyone who is trying to log into an application that has SSO enabled

3. Which affiliations and groups are used for access control?
There are neither affiliations, nor groups related to access control.  SSO itself provides access control in the form of authentication.

4. What is the process for requesting the access?
End user access to web-sso is provided upon registration (usually self-registation) to the Enterprise Person Registry

5. Levels of Authentication

Web-SSO provides two levels of authentication

  • Level 2 - username and password. Provides the minimum level of authentication requiring username and password to access a resource.
  • Level 3 - two-step verification. Provides additional security by requiring a user to login by entering a one-time token in addition to username and password.  Users are required to register for two factor authentication through the mytoken service.
  • Note: Level 0 and Level 1 are deprecated levels of authentication.

6. Desktop Single Signon (DSSO)

A Desktop Single Sign On solution has been developed to provide a seamless web-sso authentication solution for domain-connected users running the DSSO service.  Once logged onto their computer, users are able to access web-sso protected resources without having to manually authenticate.
